# 📖 Login Endpoint: Verifying Passwords & Issuing JWT

Now that we understand the theory behind JWT, it's time to build the "Identity Factory"—the endpoint that verifies credentials and hands out digital badges.

---

### 🗝️ Introduction
We have the models, we have the hashed passwords, and we have the concept of JWT. This lesson bridges the gap by implementing the actual behavior of a Login system in a real-world ASP.NET Core environment.

### 🎯 Goal
To create a dedicated `AuthController` that verifies a student's email and password securely and returns a signed JWT if they are legitimate.

---

### 🔹 The Golden Design Rule: Separation of Concerns
One of the most common mistakes is putting login logic inside the `StudentsController`. 
* **StudentsController:** Manages student data (CRUD).
* **AuthController:** Manages identity, tokens, and sessions.
📌 Keep them separate to maintain a clean, professional architecture.

---

### 🧱 Recommended Project Structure
StudentApi
│
├── Controllers
│   ├── StudentsController.cs   (Student data only)

### 🛠️ Implementation Step-by-Step

#### 1️⃣ The Login Request (DTO)
We create a simple class to capture the user's input without exposing our full Student model.

    // Models/LoginRequest.cs
    public class LoginRequest {
        public string Email { get; set; }
        public string Password { get; set; }
    }

#### 2️⃣ The AuthController (The Logic)
This controller handles the POST request to `/api/auth/login`.

    [ApiController]
    [Route("api/[controller]")]
    public class AuthController : ControllerBase {
        [HttpPost("login")]
        public IActionResult Login([FromBody] LoginRequest request) {
            // 1. Find Student
            var student = StudentDataSimulation.StudentsList.FirstOrDefault(s => s.Email == request.Email);
            if (student == null) return Unauthorized("Invalid credentials");

            // 2. Verify Password (BCrypt)
            bool isValid = BCrypt.Net.BCrypt.Verify(request.Password, student.PasswordHash);
            if (!isValid) return Unauthorized("Invalid credentials");

            // 3. Create Identity Claims
            var claims = new[] {
                new Claim(ClaimTypes.NameIdentifier, student.Id.ToString()),
                new Claim(ClaimTypes.Email, student.Email),
                new Claim(ClaimTypes.Role, student.Role)
            };

            // 4. Sign and Generate JWT
            var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes("THIS_IS_A_VERY_SECRET_KEY_123456"));
            var creds = new SigningCredentials(key, SecurityAlgorithms.HmacSha256);
            var token = new JwtSecurityToken(
                issuer: "StudentApi",
                audience: "StudentApiUsers",
                claims: claims,
                expires: DateTime.Now.AddMinutes(30),
                signingCredentials: creds
            );

            return Ok(new { token = new JwtSecurityTokenHandler().WriteToken(token) });
        }
    }

---

### 📌 Critical Notes
* **NuGet Packages:** You must install `System.IdentityModel.Tokens.Jwt` and `Microsoft.IdentityModel.Tokens` via NuGet to handle token generation.
* **Secret Key:** In a real app, the secret key should be stored in `appsettings.json` or Environment Variables, never hardcoded in the controller.

---

### ❌ Common Beginner Mistakes
* **Detailed Errors:** Returning "User not found" or "Wrong password". **NEVER** do this. Always return a generic "Invalid credentials" to prevent hackers from harvesting valid emails.
* **Missing BCrypt:** Comparing the plain-text password directly to the hash. You must use `BCrypt.Verify()`.
* **Incorrect Order:** Mapping the login endpoint inside a controller that requires authorization.

---

### 🧬 Characteristics
* **Stateless:** No data is stored on the server after the token is sent.
* **Secure:** Passwords are never stored in plain text; tokens are cryptographically signed.
* **Role-Based:** The token carries the student's role for future access control.

---

### 🔗 Interconnection
* **BCrypt:** Verifies that the "Who" matches the stored "PasswordHash".
* **JWT:** Packages that "Who" into a portable, signed format.
* **Claims:** These pieces of identity will be used by the **Authorization Middleware** in the next lesson to allow or block access.
