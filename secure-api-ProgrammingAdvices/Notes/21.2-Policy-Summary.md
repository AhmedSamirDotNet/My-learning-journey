# 🧱 تطبيق الـ Policies في الـ Student API: الربط بين الملكية وصلاحيات المدير

### 🗝️ مقدمة (Introduction)

في الدروس اللي فاتت، اتعلمنا إزاي نكريت Policy مبنية على الملكية (Ownership) ونخلي المنطق بتاعها في مكان واحد بعيد عن الكنترولر. دلوقتي جه وقت التنفيذ العملي.
الأمان مش بس إنك تكتب قواعد، الأمان هو إنك تطبقها في كل مكان صح.

---

### 1️⃣ جدول حدود الأمان (The Authorization Boundary Table)

ده "العقد الأمني" بتاع الـ API بتاعك. كل Endpoint لازم تتبع قاعدة معينة:

| الـ Endpoint                | مستوى الوصول (Access)       | نوع القاعدة                     |
| :-------------------------- | :-------------------------- | :------------------------------ |
| `GET /api/Students/Passed`  | عام (Public)                | [AllowAnonymous]                |
| `GET /api/Students/Average` | عام (Public)                | [AllowAnonymous]                |
| `GET /api/Students/{id}`    | **صاحب البيانات أو المدير** | **Policy: StudentOwnerOrAdmin** |
| `GET /api/Students/All`     | المدير فقط                  | [Authorize(Roles = "Admin")]    |
| `POST /api/Students`        | المدير فقط                  | [Authorize(Roles = "Admin")]    |
| `PUT /api/Students/{id}`    | المدير فقط                  | [Authorize(Roles = "Admin")]    |
| `DELETE /api/Students/{id}` | المدير فقط                  | [Authorize(Roles = "Admin")]    |

---

### 2️⃣ إمتى نستخدم Policy وإمتى نستخدم Role؟

- **استخدام الـ Roles:** بنستخدمها في العمليات الإدارية العامة اللي مش مرتبطة بشخص معين (زي إن المدير يشوف كل الطلاب أو يمسح طالب).
- **استخدام الـ Policies:** بنستخدمها لما القرار يعتمد على داتا حية (Runtime Data). زي إن طالب عايز يشوف بروفايله، هنا القاعدة بتقول: "لازم تكون إنت صاحب الـ ID ده أو تكون Admin".

---

### 3️⃣ التنفيذ العملي: تطبيق الـ Ownership Policy

داخل الـ `GetStudentById` أكشن، مش هنكتب `if` يدوي للتشييك على الـ Role، هنسيب الـ Policy هي اللي تحكم.

[HttpGet("{id}")]
public async Task<ActionResult<Student>> GetStudentById(int id, [FromServices] IAuthorizationService authService)
{
// 1. نجيب الطالب من قاعدة البيانات (أو الداتا اللي عندنا)
var student = StudentDataSimulation.StudentsList.FirstOrDefault(s => s.Id == id);
if (student == null) return NotFound();

    // 2. هنا "القاضي" (Policy) بيحكم بناءً على الـ ID اللي جاي في الـ URL
    var authResult = await authService.AuthorizeAsync(User, id, "StudentOwnerOrAdmin");

    if (!authResult.Succeeded)
        return Forbid(); // هيرجع 403 Forbidden لو مش صاحبه ولا Admin

    return Ok(student);

}

---

### 🧪 4️⃣ اختبار سلوك الأمان (Testing)

عشان نتأكد إن السيستم شغال صح، لازم نجرب 3 حالات:

1. **طالب بيدخل على بياناته:** لو طالب ID بتاعه 5 وطلب `/api/Students/5` -> لازم يرجع **200 OK**.
2. **طالب بيدخل على بيانات غيره:** لو طالب ID بتاعه 5 وطلب `/api/Students/10` -> لازم يرجع **403 Forbidden**.
3. **المدير (Admin):** لو دخل على أي ID -> لازم يرجع **200 OK**.

---

### 💡 ليه الهيكلية دي "بتكبر" معاك (Scalable)؟

- **تقليل التكرار:** المنطق مكتوب في الـ Handler مرة واحدة بس.
- **سهولة المراجعة (Audit):** تقدر تشوف كل قواعد الأمان في فولدر واحد (Authorization).
- **مرونة التعديل:** لو قررت بكرة إن "المشرف" كمان يقدر يشوف البيانات، هتعدل الـ **Handler** بس، وكل الـ Endpoints هتتحدث أوتوماتيك.

---

### 🔗 الربط النهائي (The Big Picture)

- **JWT:** بتقول إنت مين (Identity).
- **Roles:** بتقول رتبتك إيه في السيستم (Rank).
- **Policies:** بتقول هل البيانات دي تخصك ولا لأ (Data Ownership).
- **Controller:** بيفضل نظيف ومهمته بس تنفيذ الشغل (Business Logic).

---

### 🛠️ ملخص نهائي

الـ Roles بتحمي الـ **Actions** (مين يقدر يمسح؟)، والـ Policies بتحمي الـ **Data** (بيمسح داتا مين؟). الربط بينهم هو اللي بيعمل API احترافي ضد أي اختراق.
