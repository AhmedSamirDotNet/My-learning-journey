# 🛡️ المرجع التقني: الـ Slow Hashing وعملية التكرار (Iterations)

هذا المستند يشرح الميكانيكا الداخلية لكيفية حماية كلمات المرور من الهجمات العنيفة (Brute-Force) باستخدام تقنية الـ Key Stretching.

---

### 1. ما هي اللفة (Iteration) تقنياً؟

اللفة ليست مجرد إعادة تنفيذ، بل هي عملية **تراكمية تسلسلية (Recursive)**. ناتج اللفة السابقة يصبح هو "المدخل" للفة التالية.

**المعادلة المنطقية:**
$$Hash_{final} = Hash(Hash(Hash(...Hash(Password + Salt)...)))$$

**خطوات التنفيذ داخل المعالج (CPU):**

1. **Initial Step:** دمج كلمة المرور مع الملح عشوائي (Salt).
2. **Round 1:** تمرير الخليط لخوارزمية الهاش (مثل SHA-256) والحصول على ناتج أولى.
3. **Round 2:** أخذ "ناتج" الجولة الأولى وإدخاله كمدخل وحيد للخوارزمية مرة أخرى.
4. **Round 3:** أخذ ناتج الجولة الثانية وتكرار العملية.
5. **Final Step:** بعد الوصول للعدد المطلوب (مثلاً 600,000 لفة)، يتم تخزين الناتج الأخير فقط.

---

### 2. لماذا نستخدم آلاف اللفات؟ (الهدف الأمني)

الهدف هو القضاء على تفوق الهاردوير لدى المخترق (**Hardware Acceleration Resistance**).

- **المخترق:** يستخدم كروت شاشة (GPUs) أو أجهزة (ASIC) قادرة على حساب ملايين الهاشات في الثانية.
- **الحل:** بما أن كل لفة تعتمد على نتيجة ما قبلها، يضطر جهاز المخترق إلى تنفيذها **بالتتابع** (Sequentially). لا يمكنه حساب اللفة رقم 100 قبل اللفة رقم 99. هذا يجعل محاولة تخمين كلمة مرور واحدة تأخذ وقتاً طويلاً جداً، مما يجعل الهجوم "غير مجدٍ اقتصادياً".

---

### 3. مقارنة الخوارزميات (Technical Benchmark)

| الخوارزمية | الميكانيكا                                                 | نقاط القوة                                                    | الاستخدام الموصى به                                      |
| :--------- | :--------------------------------------------------------- | :------------------------------------------------------------ | :------------------------------------------------------- |
| **PBKDF2** | تكرار بسيط للهاش.                                          | سهولة الإعداد وتوافقية عالية.                                 | جيدة، لكنها بدأت تضعف أمام الـ GPUs الحديثة.             |
| **BCrypt** | تعتمد على خوارزمية Blowfish مع توسيع المفتاح بطريقة بطيئة. | مقاومة ممتازة لكروت الشاشة.                                   | **المعيار الذهبي** حالياً لأغلب تطبيقات الويب والـ APIs. |
| **Argon2** | تعتمد على المعالج (CPU) والذاكرة (RAM) معاً.               | الأقوى على الإطلاق؛ تستهلك ذاكرة السيرفر لمنع الهجمات الضخمة. | **الأفضل للمستقبل** وللأنظمة شديدة الحساسية.             |

---

### 4. كيف تختار عدد اللفات المناسب؟ (Tuning)

الرقم ليس ثابتاً، بل يتناسب مع قوة سيرفرات شركتك:

- **المعيار:** يجب أن تستغرق عملية التحقق من كلمة المرور ما بين **200ms إلى 500ms** على سيرفر الإنتاج (Production).
- إذا كان السيرفر جباراً، زد عدد اللفات لزيادة الأمان.
- إذا كان السيرفر ضعيفاً، قلل اللفات بما لا يقل عن الحدود الدنيا الموصى بها عالمياً (مثلاً 600,000 لـ PBKDF2).

---

### 5. مثال كود تخيلي (Pseudo-code)

لتبسيط اللوجيك البرمجي:

```csharp
// مدخلات العملية
string password = "User_Password_123";
string salt = "Random_Salt_Generated_Per_User";
int iterations = 100000;

// أول عملية هاش
byte[] result = Hash(password + salt);

// حلقة التكرار (The Loop)
for (int i = 1; i < iterations; i++)
{
    // سحب الناتج القديم وإعادة ضخمه في الخوارزمية
    result = Hash(result);
}

// النتيجة النهائية التي تحفظ في الـ Database
SaveToDatabase(result);
```
