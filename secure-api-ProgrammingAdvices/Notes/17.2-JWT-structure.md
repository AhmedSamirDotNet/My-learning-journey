# 🎫 Deep Dive: JSON Web Token (JWT) Structure

يتكون الـ JWT من ثلاثة أجزاء رئيسية مفصولة بنقطة `.` (Base64Url Encoded)، ويكون شكله النهائي هكذا: `Header.Payload.Signature`

---

### 1️⃣ الجزء الأول: الـ Header (الرأس)

هو الجزء المسؤول عن تعريف "نوع" التوكن و"الخوارزمية" المستخدمة في التشفير.

- **المحتوى:** يتكون عادةً من حقلين:
  1. `alg`: الخوارزمية المستخدمة (مثل HS256).
  2. `typ`: نوع التوكن (دائماً JWT).
- **مثال:**
  {
  "alg": "HS256",
  "typ": "JWT"
  }

---

### 2️⃣ الجزء الثاني: الـ Payload (الحمولة / البيانات)

هذا هو قلب التوكن، حيث يحتوي على المعلومات (Claims) التي نريد نقلها (مثل بيانات المستخدم). تنقسم الـ Claims إلى ثلاثة أنواع:

1. **Registered Claims:** بيانات محددة مسبقاً من قبل المعايير الدولية (اختيارية لكن يُنصح بها):
   - `iss` (Issuer): من أصدر التوكن.
   - `exp` (Expiration Time): وقت انتهاء الصلاحية.
   - `sub` (Subject): معرف المستخدم (UserID).
2. **Public Claims:** بيانات عامة يمكن تعريفها (مثل اسم المستخدم).
3. **Private Claims:** بيانات مخصصة يتم الاتفاق عليها بين السيرفر والفرونت إند (مثل الـ Roles).

⚠️ **تحذير أمني:** هذا الجزء "مكشوف" ويمكن لأي شخص فكه وقراءته بسهولة (Base64)، لذا **لا تضع فيه كلمات سر أو بيانات حساسة جداً**.

---

### 3️⃣ الجزء الثالث: الـ Signature (التوقيع)

هو الجزء الأكثر أهمية، وظيفته التأكد من أن التوكن لم يتم التلاعب به.

- **كيف يتم إنشاؤه؟**
  يأخذ السيرفر (Header + Payload) ويقوم بدمجهم مع "مفتاح سري" (Secret Key) لا يعرفه أحد غير السيرفر، ثم يمررهم على خوارزمية التشفير المذكورة في الـ Header.
- **المعادلة:**
  HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), secret_key)

---

### 🔄 كيف يتحقق السيرفر من صحة التوكن؟

عندما يرسل المستخدم التوكن، يقوم السيرفر بالخطوات التالية:

1. يستلم (Header + Payload) من التوكن القادم.
2. يستخدم الـ **Secret Key** المخزن لديه ليصنع "توقيعاً جديداً" من هذه البيانات.
3. يقارن التوقيع الجديد بالتوقيع الموجود في التوكن (Signature).
4. **النتيجة:**
   - إذا تطابقا: السيرفر يتأكد أن البيانات سليمة (Valid).
   - إذا اختلفا: السيرفر يعرف أن شخصاً ما حاول تعديل الـ Payload (مثلاً حاول تغيير دوره من User إلى Admin)، فيرفض الطلب فوراً.

---

## Correct JWT trust flow order is:

- Read token → Check signature → Check expiration → Extract claims

### 🧬 ملخص الأمان

- **Encrypted vs Encoded:** الـ JWT يتم عمل Encoding له وليس Encryption (أي أنه قابل للقراءة ولكن غير قابل للتزوير).
- **Stateless:** السيرفر لا يحتاج لعمل Query للـ DB للتأكد من الجلسة، يكفي فقط التحقق من التوقيع (Signature Verification).
