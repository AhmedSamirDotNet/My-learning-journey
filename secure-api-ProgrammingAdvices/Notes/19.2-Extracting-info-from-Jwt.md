# 🧭 The Journey of a JWT: From String to ClaimsPrincipal

### 🔄 المرحلة الأولى: شكل التوكن وهي "مسافرة" (The Token String)

بعد عملية الـ Login، تخرج التوكن كـ نص مشفر (String) مكون من 3 أجزاء مفصولة بنقاط:

1. **Header:** يحتوي على نوع التوكن وخوارزمية التشفير.
2. **Payload:** يحتوي على الـ Claims (البيانات التي تم تخزينها مثل ID والـ Role).
3. **Signature:** الختم الأمني الذي يضمن عدم التلاعب بالبيانات.

- **المسار:** يحمل المستخدم هذا النص ويضعه في الـ `Authorization Header` كـ `Bearer <token>` مع كل طلب.

---

### 🏛️ المرحلة الثانية: الأوبجكت الأصلي (The ClaimsPrincipal Hierarchy)

عندما يستلم الـ API التوكن، يقوم بتحويلها برمجياً إلى كائن هرمي يسمى **ClaimsPrincipal**. إليك تركيبه من الداخل للخارج:

1. **المستوى الأول (الغلاف) - ClaimsPrincipal:**
   - هو الذي نصل إليه في الكود بكلمة `User`.
   - يمثل "الحاوية" (Container) التي تضم كل هويات المستخدم.
2. **المستوى الثاني (البطاقة) - ClaimsIdentity:**
   - هو خاصية داخل الـ Principal تسمى `Identities`.
   - يمثل "البطاقة الشخصية" المستخرجة من التوكن (يمكن للمستخدم امتلاك أكثر من هوية).
3. **المستوى الثالث (السطور) - Claims:**
   - قائمة موجودة داخل الـ Identity.
   - كل سطر (Claim) يمثل معلومة محددة عبارة عن (Key/Value).

---

### ⚙️ المرحلة الثالثة: إزاي الأوبجكت ده اتملا؟ (The Middleware Logic)

تتم هذه العملية آلياً بواسطة الـ Middleware في الخلفية فور وصول الطلب:

- **الخطوة 1 (الاستلام):** يقرأ الـ API قيمة الـ `Authorization Header`.
- **الخطوة 2 (الفك):** يقوم الـ `JwtBearerMiddleware` باستخدام الـ "Secret Key" لفك تشفير الـ Payload والتأكد من صحة التوقيع.
- **الخطوة 3 (التعبئة - Hydration):** - يتم إنشاء أوبجكت `ClaimsIdentity`.
  - يتم عمل `foreach` على كل معلومة في الـ Payload (مثل ID, Role, Email).
  - يتم تحويل كل معلومة إلى `new Claim(...)` وإضافتها لقائمة الـ Identity.
- **الخطوة 4 (التثبيت):** يتم وضع الـ Identity داخل `ClaimsPrincipal` جديد وربطه بالـ `HttpContext` ليصبح متاحاً طوال دورة حياة الطلب.

---

### 🛠️ المرحلة الرابعة: الاستخدام جوه الكنترولر (The Access)

الآن يمكنك الوصول للبيانات داخل الـ Action بكل سهولة:

- **لماذا نستخدم `User`؟**
  لأنه يمثل الـ `ClaimsPrincipal` الذي قام الـ Middleware بتجهيزه وحقنه بالبيانات المستخلصة من التوكن.
- **لماذا نستخدم `FindFirstValue`؟**
  بما أن الـ `User` يحتوي على قائمة من الـ Claims، فنحن نطلب منه: "ابحث في القائمة (Find) عن أول سطر (First) من نوع معين (مثل NameIdentifier) وأعطني القيمة النصية الموجودة بداخله (Value)".

---

### 🔗 ملخص الترابط (The Chain)

التوكن (نص) ⬅️ الميدل وير (يفك النص) ⬅️ إنشاء Identities ⬅️ وضعها في Principal (اسمه User) ⬅️ البحث (Find) داخل الـ User لجلب الـ ID.

# 🧭 Ownership-Based Authorization: Claims Injection & Extraction

### أولاً: مرحلة الـ "تعبئة" (The Login Phase) 📥

في هذه المرحلة، نقوم بتجهيز الـ Claims التي ستحملها التوكن "في حقيبتها" لتسافر بها إلى المتصفح. سنقوم بإضافة أنواع قياسية (Standard) وأنواع مخصصة (Custom).

**الكود البرمجي (Implementation):**

[HttpPost("login")]
public IActionResult Login([FromBody] LoginRequest request)
{
var student = StudentDataSimulation.StudentsList
.FirstOrDefault(s => s.Email == request.Email);

    if (student == null || !BCrypt.Net.BCrypt.Verify(request.Password, student.PasswordHash))
        return Unauthorized("Invalid credentials");

    // 1. تحديد الـ Claims (المعلومات اللي هتمشي في التوكن)
    var claims = new List<Claim>
    {
        // النوع القياسي للـ ID
        new Claim(ClaimTypes.NameIdentifier, student.Id.ToString()),

        // النوع القياسي للـ Email
        new Claim(ClaimTypes.Email, student.Email),

        // النوع القياسي للـ Role
        new Claim(ClaimTypes.Role, student.Role),

        // --- إضافة Custom Claims من عندنا ---
        new Claim("FullName", student.Name), // معلومة إضافية مخصصة
        new Claim("Organization", "TechTober Systems"), // اسم شركتك
        new Claim("IsPremium", "true") // معلومة تستخدم في الـ Authorization اللاحق
    };

    var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes("THIS_IS_A_VERY_SECRET_KEY_123456"));
    var creds = new SigningCredentials(key, SecurityAlgorithms.HmacSha256);

    // 2. تكريت التوكن وحقن الـ Claims فيها (Payload)
    var token = new JwtSecurityToken(
        issuer: "StudentApi",
        audience: "StudentApiUsers",
        claims: claims,
        expires: DateTime.Now.AddMinutes(30),
        signingCredentials: creds
    );

    return Ok(new { token = new JwtSecurityTokenHandler().WriteToken(token) });

}

### ثانياً: مرحلة الـ "استخراج" (The Request Phase) 📤

بمجرد وصول الـ Request ومعه التوكن، يقوم الـ Middleware بفكها وتعبئة أوبجكت الـ `User` تلقائياً. إليك كيف ننادي هذه البيانات داخل الكنترولر:

**الكود البرمجي (Usage):**

[HttpGet("{id}")]
public ActionResult<Student> GetStudentById(int id)
{
// 1. استخراج الـ ID (الـ Claim القياسي)
// نستخدم FindFirstValue لأن User عبارة عن فهرس، نبحث فيه عن أول قيمة مطابقة
var userId = User.FindFirstValue(ClaimTypes.NameIdentifier);

    // 2. استخراج الـ Role (الـ Claim القياسي)
    var userRole = User.FindFirstValue(ClaimTypes.Role);

    // 3. استخراج الـ Custom Claims التي أضفناها يدوياً
    var orgName = User.FindFirstValue("Organization"); // سيُرجع "TechTober Systems"
    var isPremium = User.FindFirstValue("IsPremium");  // سيُرجع "true"

    // 4. فحص سريع للصلاحية (Built-in check)
    if (User.IsInRole("Admin")) { /* صلاحيات المدير */ }

    // --- تطبيق منطق الـ Ownership (المقارنة) ---
    int authenticatedStudentId = int.Parse(userId);

    // إذا لم يكن المدير وكان يحاول الوصول لـ ID غير الـ ID الخاص به
    if (userRole != "Admin" && authenticatedStudentId != id)
        return Forbid(); // 403 Forbidden

    var student = StudentDataSimulation.StudentsList.FirstOrDefault(s => s.Id == id);
    return student == null ? NotFound() : Ok(student);

}

### ثالثاً: معلومات "تريكّات" تقنية (Pro-Tips) 💡

1. **سر الـ FindFirstValue:**
   لماذا نستخدم `Find`؟ لأن التوكن قد تحمل أكثر من Role لنفس المستخدم (مثلاً: Admin و Manager). الـ `FindFirstValue` تجلب أول تطابق، بينما `FindAll` تجلب القائمة كاملة.

2. **لماذا ClaimTypes وليست "id"؟**
   `ClaimTypes.NameIdentifier` هو ثابت (Constant) يحمل رابطاً قياسياً (URI). الإطار البرمجي (Framework) يفضل هذه الأنواع القياسية لكي تعمل الوظائف الجاهزة مثل `User.IsInRole` بشكل سليم دون إعدادات إضافية.

3. **أين اتملا الـ User؟**
   يحدث "السحر" في الـ Middleware المسمى `UseAuthentication()`. هو من يأخذ التوكن، يفك الختم (Signature)، ويحول كل سطر في الـ Payload إلى `Claim Object` يضعه داخل الـ `User`.

4. **تحذير الأمان (Critical):**
   أي معلومة توضع في الـ Claims (مثل FullName) هي معلومات **Encoded** وليست **Encrypted**. أي شخص يفك التوكن (Base64) سيراها بوضوح. لذا، **إياك وضع كلمات سر أو بيانات حساسة جداً** داخل الـ Claims. هي مخصصة لبيانات الـ Authorization فقط.

---

### 🔗 الخلاصة:

في الـ **Login**، نقوم بـ "حشر" البيانات في قائمة الـ Claims لنرسلها في التوكن. وفي الـ **Controller**، نقوم بـ "سحب" هذه البيانات من أوبجكت الـ `User` الجاهز، لأن الـ `User` ما هو إلا مرآة برمجية لكل ما كان موجوداً داخل التوكن النصية.
